# 🤖 Agent's Game Dev Guide (실전 개발 노하우)

이 가이드는 AI 에이전트와 개발자가 추첨 게임 포털의 게임을 수정하거나 새롭게 만들 때, **"어떻게 하면 저사양 PC에서도 부드럽고, 박진감 넘치는 게임을 만들 수 있는가"**에 대한 실전 경험을 정리한 문서입니다.

---

## 1. 일관된 게임 속도: 델타 타임 (Delta Time)
프레임 기반(`requestAnimationFrame` 호출마다 일정 거리 이동)으로 구현하면 성능이 좋은 PC에서는 너무 빠르고, 느린 PC에서는 너무 느려집니다. 하드웨어 사양에 상관없이 동일한 물리 법칙을 적용하려면 반드시 **델타 타임**을 사용하세요.

### 델타 타임 적용 패턴
```javascript
let lastTimestamp = 0;

function gameLoop(timestamp) {
  if (!gameRunning) return;
  
  // 초 단위의 차이 계산 (60fps 기준 약 0.016s)
  if (!lastTimestamp) lastTimestamp = timestamp;
  const dt = (timestamp - lastTimestamp) / 1000;
  lastTimestamp = timestamp;

  // 탭 전환 등으로 인한 급격한 튀튀 방지
  const cappedDt = Math.min(dt, 0.1);
  
  update(cappedDt);
  draw();
  requestAnimationFrame(gameLoop);
}

function update(dt) {
  const dtScale = dt * 60; // 60fps 기준 보정값
  // 위치 이동 시 dtScale을 곱해줌
  player.x += player.vx * dtScale;
  gravity = 0.1 * dtScale;
}
```

## 2. 성능 최적화: 캔버스 캐싱 (Canvas Caching)
매 프레임마다 `ctx.arc()`, `ctx.stroke()`, `ctx.fill()`을 수십 번 호출하는 것은 저사양 기기에서 매우 치명적입니다. 변하지 않는 배경이나 반복되는 복잡한 오브젝트는 가상 캔버스에 미리 그려두세요.

### 최적화 전략
- **정적 배경**: 하늘, 나무, 잔디 등은 게임 시작 시 또는 리사이즈 시에만 `treeCanvas`에 한 번 그리고, `draw` 함수에서는 `ctx.drawImage(treeCanvas, 0, 0)` 한 줄만 실행합니다.
- **오브젝트 스프라이트**: 가시가 많은 밤송이처럼 복잡한 이모지/도형은 `chestnutCache` 캔버스에 미리 렌더링한 후 이미지를 복사해 사용하세요. (실제 '밤송이 서바이벌'에서 성능이 5배 이상 향상됨)

## 3. 박진감 넘치는 연출 (Game Juice)
추첨 게임은 결과만큼이나 '과정의 긴장감'이 중요합니다. 아래 효과들을 적극 활용하세요.

- **스크린 셰이크 (Screen Shake)**: 캐릭터가 충돌하거나 폭발할 때 게임 영역 전체를 짧게 흔들어주세요. (CSS Animation이나 Canvas Transform 활용)
- **반동 효과 (Recoil)**: 물건을 받거나 충격이 가해질 때, 물체가 날아온 반대 방향으로 살짝 밀렸다가 돌아오는 `cubic-bezier(0.175, 0.885, 0.32, 1.275)` 곡선을 활용하세요.
- **포물선 던지기**: `Web Animations API`를 사용하여 폭탄이나 물체를 던질 때 중간 지점에서 `scale`을 키우면 3D 포물선 느낌을 줄 수 있습니다.
- **방향 인지**: 폭탄을 들 때 단순히 위에 띄우지 말고, 원의 중심을 향해 '앞으로 내밀고 있는' 위치를 계산(벡터 활용)하여 배치하세요.

## 4. 난이도 가속 (Escalation)
게임이 처음부터 끝까지 같은 속도면 지루합니다.
- **시간 기반**: 게임 시작 10초 후부터 중력이나 속도를 `difficultyMultiplier`를 통해 서서히 높이세요.
- **상태 기반**: 생존자가 줄어들수록(탈락자가 늘어날수록) 남은 이들의 속도를 높여 긴장감을 유도하세요.
- **UI 피드백**: 가속 모드가 시작될 때 화면에 경고 문구(예: `⚠️ 위험! 가속 모드 ⚠️`)를 띄워 플레이어에게 시각적 신호를 줍니다.

## 5. 반응형 게임 레이아웃
브라우저 창 크기에 따라 게임 요소가 너무 작아지지 않게 관리하세요.
- **CSS 변수 활용**: `:root`에 `--circle-size`, `--player-size` 등을 정의하고 미디어 쿼리에서 이 변수값만 조정합니다.
- **리사이즈 대응**: `window.onresize` 시점에 물리 좌표를 다시 계산하여 캐릭터들이 화면 밖으로 나가지 않게 재배치합니다.

### 읽기 쉬운 폰트 기준
- **승인된 기준 화면**: `dart-tournament/index.html`의 2026-07-18 타이포그래피를 포털 게임의 가독성 기준으로 삼습니다. 새 게임이나 UI 개편 시 이 화면과 체감 크기를 비교하세요.
- **HTML UI 권장 크기**: 본문·입력·버튼은 대체로 `0.8rem ~ 0.9rem`, 보조 라벨은 `0.65rem ~ 0.7rem`, 주요 화면 제목은 데스크톱에서 `1.25rem` 이상을 기준으로 합니다. 작은 영문 장식 문구를 제외한 핵심 정보는 `0.6rem` 아래로 축소하지 마세요.
- **Canvas 글자 보정**: 캔버스는 화면에 맞춰 축소되므로 논리 좌표상의 `10px ~ 12px` 글자는 실제 화면에서 지나치게 작아집니다. `1200×800` 논리 캔버스 기준으로 보조 라벨 `13px ~ 15px`, 선수명 `32px`, 주요 점수 `58px`, 과녁 숫자 `24px` 정도를 출발점으로 사용하세요.
- **실제 크기 검증**: CSS 수치만 확인하지 말고 최소한 데스크톱 `1440×900`과 모바일 `390×844`에서 직접 렌더링하여 줄바꿈, 잘림, 가독성을 함께 확인하세요. 모바일은 별도 미디어 쿼리로 조정하되 핵심 정보의 가독성을 희생하지 마세요.

### 사이드바 랭킹과 하단 버튼 기준
- **승인된 기준 화면**: `chaser-survival/index.html`의 사이드바 하단 구조를 공통 기준으로 삼습니다.
- **랭킹 영역 확장**: 데스크톱 사이드바는 세로 `flex` 컨테이너로 구성하고 LIVE SCOREBOARD에 `flex: 1; min-height: 0`을 적용하여 남은 공간을 모두 채우세요. 랭킹 목록에는 `flex: 1; min-height: 0; overflow-y: auto`를 적용하고 데스크톱에서 임의의 `max-height`로 공간을 제한하지 마세요.
- **버튼을 최하단에 고정**: 하단 내비게이션은 `flex-shrink: 0`으로 랭킹 아래에 배치하여 항상 사이드바의 최하단에 보여야 합니다. 모바일처럼 사이드바 높이가 자동인 화면에서만 랭킹에 적절한 `max-height`와 내부 스크롤을 사용하세요.
- **표준 버튼 구성**: 왼쪽부터 포털 홈, 결과 복사, GitHub 저장소, 커피 후원의 네 버튼을 사용합니다. 의미가 모호한 유니코드 기호 대신 `chaser-survival`과 같은 명확한 SVG 아이콘을 사용하세요.
- **접근성과 외부 링크**: 각 버튼에 기능을 설명하는 `aria-label`과 툴팁을 제공하고, 외부 링크에는 `target="_blank"`와 `rel="noopener noreferrer"`를 함께 지정하세요.

## 6. 데이터 신뢰성
- **localStorage**: 모든 설정값과 참여자 명단은 입력 즉시 저장하여 새로고침 시에도 유지되어야 합니다.
- **정밀 타이머**: 초 단위만 보여주더라도 내부적으로는 소수점 셋째 자리(`.3f`)까지 계산하여 '동시에 죽었을 때'의 순위 정렬이나 기록 경쟁에 활용하세요.

## 7. 시스템 결정론 (Deterministic Seeding)
추첨 게임에서 가장 중요한 것은 **공정성**과 **재현성**입니다.
- **해시 기반 RNG**: SHA-256 등의 해시 알고리즘을 사용하여 입력된 시드(참여자 명단 포함)로부터 고정된 난수 시퀀스를 생성하세요. "누가 이겨도 이상할 게 없는 결과"가 아니라 "동일 시드면 항상 동일 결과"가 나와야 합니다.
- **시드 공유**: 대기 화면에 사용된 해시값(`HASH: ABCD...`)을 표시하면 시뮬레이션의 신뢰도를 크게 높일 수 있습니다.

## 8. 승리 연출의 미학 (Cinematic Results)
결과 화면은 게임의 '클라이맥스'입니다. 단순한 텍스트 노출을 넘어 시네마틱한 경험을 제공하세요.
- **글래스모피즘 & 블러**: `backdrop-filter: blur(20px)`를 활용한 전체 화면 오버레이로 게임의 흔적을 남기면서 결과에 집중하게 만듭니다.
- **Celebration Layer**: 메인 게임 캔버스와 분리된 별도의 `celebCanvas`를 사용하여 화려한 파티클/폭죽 연출을 줌에도 메인 카메라 연출(줌, 이동)과 충돌하지 않게 설계하세요.
- **1초의 정적 (Suspense)**: 최종 우승자가 결정된 즉시 결과창을 띄우지 말고, 약 1초 정도의 딜레이를 주어 사용자가 결과의 여운을 느낄 수 있게 하세요.

## 9. 지능형 카메라 (Intelligent Camera)
시뮬레이션의 범위가 가변적일 때 유용합니다.
- **액션 존 트래킹**: 살아남은 모든 입자를 포함하는 최소 사각형(Bounding Box)을 계산하여 카메라의 `zoom`과 `offset`을 실시간으로 조정하세요.
- **One-Way Zoom**: 카메라가 앞뒤로 어지럽게 움직이는 '울렁거림'을 방지하기 위해, 한 번 확대된 줌은 이전보다 작아지지 않게(줌 아웃 금지) 로직을 설계하면 시각적 안정감이 크게 향상됩니다.
